# OpenClaw on Render for `carroll-john/openclaw`

This fork is set up so Codex can manage future changes through Git.

## What is already wired

- `origin` points to your fork: `https://github.com/carroll-john/openclaw.git`
- `upstream` points to the official repo: `https://github.com/openclaw/openclaw.git`
- `render.yaml` is already present at the repo root and matches the current OpenClaw Render docs:
  - `runtime: docker`
  - `plan: starter`
  - `healthCheckPath: /health`
  - persistent disk at `/data`
  - `OPENCLAW_STATE_DIR=/data/.openclaw`
  - `OPENCLAW_WORKSPACE_DIR=/data/workspace`
  - `OPENCLAW_GATEWAY_PORT=8080`
  - `OPENCLAW_GATEWAY_TOKEN` auto-generated by Render

## One-time Render deploy

Use this deploy URL so Render tracks your fork instead of the upstream repo:

`https://render.com/deploy?repo=https://github.com/carroll-john/openclaw`

In Render:

1. Create the Blueprint from your fork.
2. Keep the service as a Docker web service.
3. Keep the plan on `starter` unless you intentionally want free-tier spin-down and no persistent disk.
4. Keep the disk mount at `/data`.
5. Let Render auto-generate `OPENCLAW_GATEWAY_TOKEN`.
6. Deploy.

After the first healthy deploy:

1. Open `https://<your-service>.onrender.com/`
2. Copy the `OPENCLAW_GATEWAY_TOKEN` from Render Dashboard -> your service -> Environment
3. Paste the token into the Control UI when prompted
4. Finish OpenClaw onboarding in the UI with your model provider key

## Health checks

- Health endpoint: `https://<your-service>.onrender.com/health`
- Logs: Render Dashboard -> your service -> Logs
- Shell: Render Dashboard -> your service -> Shell
- Persistent data lives under `/data`

To create a backup from the Render shell:

```bash
openclaw backup create
```

## Future Codex workflow

Render should watch `main` on your fork. The deployment flow then becomes:

1. Codex edits this local repo.
2. Changes are committed and pushed to `origin`.
3. Render auto-deploys from the pushed commit.

Recommended daily Git commands:

```bash
git status
git fetch upstream
git log --oneline --decorate --graph --max-count=12 --all
```

When you want to bring in upstream OpenClaw updates:

```bash
git fetch upstream
git checkout main
git pull --ff-only origin main
git merge upstream/main
git push origin main
```

If you prefer PR-based changes instead of pushing directly to `main`, keep Render pointed at `main` and merge reviewed branches into `main` when ready to deploy.

## Notes

- The current English OpenClaw Render docs use the gateway token flow at `/`, not the older `SETUP_PASSWORD` + `/setup` flow.
- If you ever switch to Render free tier, remove the persistent-disk expectation from your operating model. Free tier will not preserve OpenClaw state across redeploys.
